vSphere Support Bundle Triage Tool

Rapid triage of ESXi vm-support and vCenter vc-support bundles — one HTA, no dependencies, never touches the source host.
Field Manualv1.2.0Windows / mshtaESXi & vCenterGitHub ↗

1. Overview

This tool ingests a VMware ESXi or vCenter (VCSA) support bundle and turns it into a fast, scored investigation surface for two questions: what is this node (configuration, VMs, storage) and what happened on it (logins, actions, persistence, ransomware/intrusion indicators). It runs as a single Windows HTA — no install, no runtime — and reads the bundle read-only. It never runs vm-support/vc-support and never contacts the source host.

Auto-detects the bundle type
ESXi vs vCenter is decided from the tree (etc/vmware/esx.conf vs var/log/vmware/vpxd/) with README / name-prefix fallbacks. The tab set adapts.
Single node or a whole fleet
Triage one bundle, or point at a folder of bundles (a cluster) and get a roster + cross-host correlation and the vCenter→ESXi task chain.

2. How it works

  • Extraction engine: Windows' built-in C:\Windows\System32\tar.exe (bsdtar). No 7-Zip needed for .tgz/.tar.
  • Table-of-contents first: the bundle index is read once (bsdtar -tvf, cached to toc.tsv) so the Key Files board and node summary paint without unpacking gigabytes. Individual files are extracted on demand.
  • Compressed logs: .gz inner logs decompress automatically; .zst (vCenter) needs zstd.exe — the tool probes tools\ and PATH and, if missing, offers to fetch the official build from the facebook/zstd GitHub release.
  • Deep vSAN paths: vSAN datastore paths contain : (illegal on NTFS); the tool handles bsdtar's :→_ rewrite and reads deep files via stdout to stay under the 260-char path limit.
  • Output: everything lands under _Processed\<host>\vSphere\ next to the app — TOC cache, extracted files, runinfo.json, inventory.json, CSVs and reports.
Requirements: Windows 10/11 (ships bsdtar), run the .hta from a local path (network paths trigger Windows zone restrictions on the UTF-8 reader). Analysis runs read-only against a copy of the bundle.

3. Getting started

  1. Put the .hta in a working folder (e.g. Documents\Claude\Artifacts\vSphere-Bundle-Triage\) and double-click it.
  2. Bundle field: point at the nested archive that holds the tree (ESXi <host>.tar / vCenter <host>.tgz), the outer .tgz, or an already-extracted folder. Browse… or Pick folder… help.
  3. Target hostname is required — it names the output folder. It is guessed from the bundle name; correct it if needed.
  4. Click Analyze bundle. The button turns blue when the field points at a single bundle; Triage fleet folder… turns blue when it points at a folder holding two or more bundles.

Log analysis (Logons / Actions / Timeline / Persistence / IOC) runs automatically after Analyze. Use Show log to see extraction and per-log line counts.

4. The tabs

Every table in the app keeps its header row pinned while you scroll, and columns are drag-resizable (double-click a grip to reset).

Overview

Node configuration summary + key-file coverage, the intrusion / ransomware indicator panel, and the CVE exposure panel. The stat bubbles at the bottom (Logons, Actions, Timeline, IOC hits) are clickable — they jump to the matching tab. Buttons: Re-run log analysis and Case-note report (green).

Specification

Deep node configuration parsed from the bundle. ESXi: identity (FQDN, UUID, serial / service tag, model/PSNT, management IP + ESXi Host Client URL, and the iDRAC / BMC IP when in-band IPMI exposes it), networking (VMkernel adapters, physical NICs, gateway, DNS, NTP), port groups & VLANs (standard vSwitch port groups plus distributed port groups with their VLANs, parsed from esxcfg-vswitch -l and net-dvs -l), datastores, vSAN & disk layout (disk groups, physical devices, capacity/usage — for recovery scoping), and security posture (AD join via LSASS, VIB acceptance, syslog target). vCenter: identity, management IP + vSphere Client & VAMI (:5480) URLs, SSO domain & identity sources / AD integration, networking, and managed ESXi hosts. The green Export → HTML button writes a standalone node-spec document. (The same identity fields — serial, management IP, web URLs, iDRAC — also appear on the Overview → Node configuration pane.)

Security

A graded review of patch state and recommended security controls, checked against the bundle's config captures and the parsed log evidence, scored as N of M checkable controls at best practice (a bubble on Overview jumps here; the case-note report gets a matching section). Three groups:

  • Patch state — build number vs the tracked CVE fix-builds (summarises the Overview CVE panel; advisory, verify against the VMSA).
  • Hardening controls (ESXi) — VIB acceptance, host encryption mode / execInstalledOnly / Require Secure Boot, TPM & measured boot, MOB, account lockout, password policy, shell/SSH idle timeouts, DCUI access, remote syslog, NTP. Settings checks read advanced-list -d (options changed from default), so "at default" is provable, not assumed. vCenter: SSO identity sources, rsyslog forwarding, NTP.
  • Installed VIBs (software integrity) — parses software-vib-list (Name / Version / Vendor / Acceptance / Install date) and flags the rogue-VIB persistence tells: any CommunitySupported (unsigned) package, VIBs from a vendor that isn't VMware or a known hardware OEM, and VIBs installed after the base image (highlighted if within the case window). A Third-party & non-standard VIBs table lists every non-VMware / non-Certified package with its full metadata, risk-sorted.
  • Log-corroborated controls — what the logs show actually happened: SSH logins and ESXi Shell usage observed, logons from public IPs, failed-login volume, and log-integrity gaps. Config claims and observed behaviour are graded separately, so a "disabled" control that the logs show being used still surfaces.

The green Export → HTML button writes a self-contained security report: an executive summary (posture score with fail/warn/pass counts and the top findings), patch state & CVE exposure, ransomware/intrusion indicators, the full control matrix, the third-party VIB table, and the security-relevant persistence/tampering findings — a standalone deliverable distinct from the full Case-note report.

Support bundles are targeted: a control whose source file isn't captured is graded n/a — never assumed compliant. Lockdown mode on ESXi 7+ lives in the ConfigStore, which vm-support does not capture. The checks are version-aware: patch-state verdicts select the fix line by the host's version and update level; capture-driven checks degrade to n/a on versions that lack a capture; and the "at default" claims are the documented 7.x/8.x defaults — on other versions the check still proves the setting is unchanged, but flags the default value for verification.

Network

A point-in-time snapshot from the bundle's socket + process captures (ESXi localcli network ip connection list / ps; vCenter netstat -anp / ps auwwx). Three views: listening services (with the owning process and whether the bind is loopback-only or network-facing — an all-interfaces bind from an unexpected process is the review target; a hide loopback checkbox, on by default, hides the loopback-only listeners so the network-facing ones stand out — network-facing listeners are never hidden, and the header counts what was filtered), external connections (established sessions to non-private public IPs — management plane talking to the internet), and a suspicious-process scan (attacker-tool / reverse-shell patterns: nc/socat, /tmp or /dev/shm execs, bash -i, one-liner interpreters, base64 -d, miners). These feed matching Security-tab checks. On vCenter, non-routine appliance shell commands from Linux auditd also appear in the Actions tab (routine VMware automation filtered). A Web access summary panel (vCenter) aggregates the envoy front-proxy access log — the vSphere Client / API request stream — by client source IP (with request/error counts and external-vs-private scope), and scans request paths for web-attack patterns (traversal, injection, exploit/JNDI). Below it, a Recent web requests table lists the newest individual requests (time, source IP, method, response code, path, and the SOAP operation / user-agent), with a 25 / 50 / 100 row selector, a Simplified (login/logoff) checkbox that hides the noisy API/query traffic and shows only Login/Logout events (collected across the whole captured window, not just the newest rows), a hide loopback checkbox to drop 127.0.0.1/::1 and focus on real remote clients (it filters the whole panel — the per-IP summary table and this table, with a note of how many clients/requests are hidden; the checkbox appears in both places and stays in sync), and clickable paths / open raw log links to the source file. These logs are large, so the newest ~12 MB is read from the tail; external client IPs and any attack-pattern hits raise Security checks. ESXi hosts get the equivalent — a Web / API connections panel built from rhttpproxy.log (the reverse proxy in front of the Host Client / API): client source IPs, protocol, and the accessed service, aggregated and listed newest-first, with the same hide loopback filter (most host-proxy traffic is loopback, so hiding it surfaces the real remote clients).

VMs

(ESXi) Every registered VM parsed from its .vmx: name, guest OS, vCPU, vRAM, vDisk count/names, datastore/vSAN, and running status. Click a VM to open its activity timeline — power, snapshot (take/delete/consolidate), migration (vMotion), and guest-op events parsed from the VM's vmware.log (plus rotations with incl. rotated logs). Each event shows its opID and initiator: an lro-* opID means the operation came from vCenter, and that opID joins vpxa.log/vpxd.log — the vCenter→host→VM chain. Snapshot delete/consolidate right before an encryption window is the ransomware pre-destruction tell. From the timeline, view .vmx config for the full disk paths (recovery). A Guest IP column appears only when a vCenter in the fleet supplied it (ESXi bundles do not carry guest IPs). A Guest OS dropdown (auto-populated from the inventory) narrows the table to one OS — e.g. every Windows Server VM at a glance — and combines with the free-text filter. Green Export VMs → CSV.

Key Files

A curated inventory of forensically high-value files (per bundle type): category, description, presence, size, and Open folder / Open file. Blue left-border = critical artifact; an absent critical artifact is itself a finding. Green Export inventory → CSV.

Logons

Authentication events. ESXi: auth.log SSH, hostd SOAP logins, shell.log shell-enable. vCenter: websso SSO logins (user + client IP), appliance SSH, and vpxd SAML human logins. Scored for external source IPs, failures, and privileged accounts. Case-window (from/to) filters this tab.

Login sessions (paired) view. Tick the Login sessions (paired) checkbox in the toolbar to switch from raw login events to one row per API session: login time, logout time, duration, user, source IP, session ID, and the client/user-agent. vCenter: built from vpxd SessionManager login/logout calls paired by session GUID; the user comes from the SAML token on the same operation, and the source IP is time-joined from the envoy access log — only when exactly one candidate IP sits within ±3 s (ambiguous matches are left blank with a tooltip, never guessed). ESXi: built from hostd's logged in / logged out events (the logout line carries the API-invocation count and client agent); pairing uses the opID chain, falling back to the login time the logout line itself quotes. Rows are honest about gaps: a session with no logout is still active at capture, past the window end, or a failed login — and is never hidden by the duration filter; a logout with no login means the login pre-dates the captured window. Two noise filters: hide loopback (on by default — drops sessions sourced from 127.0.0.1/::1, the appliance's own services; sessions with an unresolved IP are kept) and hide brief <N s (off by default, threshold editable — drops the 0-second automated API polls). The row count reports what each filter hid. An svc tag marks solution/service accounts. Row-click opens the full detail; CSV export and Copy-for-notes follow this view. Note: an API session is any client session — the vSphere Client, managed hosts re-authenticating, solution users — not only interactive humans; vpxd.log typically covers only a few hours, so treat the window accordingly.

Actions

Privileged operations. ESXi: shell.log command history (attacker-typed commands), the vobd audit-event stream (esx.audit.*: SSH/shell enable, account events, lockdown, non-routine firewall disables — boot-replay and patching churn filtered out; rotations always read since the live file often doesn't exist), and esxupdate software changes (image-profile/VIB add-remove batches, with recurring agent re-stages collapsed into one row). vCenter: meaningful vpxd tasks (VM lifecycle, permission/role changes, sessions) attributed to a user (nearest-preceding human login — labelled user approx), appliance sudo commands, and SSO identity events (sso-events/audit_events.log: user/group creation, membership adds, password resets — with acting user and client IP, retention far beyond websso).

Filtering: beside the text/score filters, two dropdowns narrow the table by actor and by kind — each auto-populated with the distinct values present in this bundle (e.g. actor root + kind shell command isolates one operator's typed commands). The toolbar's clear resets them.

Rotated logs: the incl. rotated logs checkbox (next to Analyze) also reads hostd.0.gz, auth.1.gz, … — weeks of extra logon/action history instead of just the live file, at the cost of decompressing each rotation. vobd rotations are always read; vpxd keeps its largest-rotation behaviour.

Timeline

Merged logons + actions, time-ordered. The Score / tags column is colour-coded, and the Kind column colours shell command red, sudo orange, vpxd/task blue. A multiselect Score filter (≥5 3-4 1-2 0) narrows to the bands you tick, and a kind dropdown (auto-populated from the bundle) shows a single event kind — e.g. only logons, or only shell commands. Case-window filters this tab.

Persistence & tampering

ESXi checks now include VM autostart (vmAutoStart.xml — a rogue VM set to auto-start survives host reboots), and the Specification's distributed port groups show their human names (from hostd/dvportgroups.xml.gz) alongside the dvportgroup IDs.

File-inspection findings (verdict: suspect / review / clean / n-a). ESXi: DCUI welcome banner (ransom-note check), rc.local.d/local.sh, remote-syslog state, VIB acceptance, SSH keys, key-log integrity. vCenter: ld.so.preload, rsyslog forwarding, admin systemd units, cron, log integrity. Bundles are targeted — items the bundle does not capture are marked "not captured", not "clean".

IOC hits

An IOC.txt (one term per line — IPs, users, hashes, filenames) placed next to the app is swept across the key logs on analysis; hits are listed here with file, line, and excerpt. Use the orange Create IOC list button (left of Analyze bundle) to build or edit it in a structured editor: IOC.csv holds Value / Type / Note / Case and is the editable source; on Save it is flattened to IOC.txt — the file this tool (and every wrapper in the family) auto-loads at launch. Hash/IPv4 values are auto-typed, and any existing plain IOC.txt is imported on first edit so nothing is lost.

5. Ransomware / intrusion indicators (Overview)

ESXi kill-chain checklist derived from the parsed data — each is TRIGGERED, clear, or n/a with evidence:

  • SSH / ESXi Shell used · external source IP on login · failed-login burst
  • Host-encryption change / recovery-key access · mass VM power-off / process kill · firewall disabled
  • Payload staged in /tmp or /scratch · download / decoder / reverse-shell · bulk VMDK/VMX manipulation
  • VIB acceptance downgraded (allows unsigned VIBs)
  • Ransom notes / encrypted files on datastore — scans vmfs/volumes for ransom-note files and known ransomware extensions (low false-positive; legitimate appliance disks are not flagged)

6. CVE exposure (Overview)

Advisory panel mapping the parsed build to high-impact vSphere CVEs, with clickable links to NVD and the VMSA advisory. Cross-checks: ESXi SLP running state (OpenSLP/ESXiArgs), AD integration (AD auth-bypass). Tracked advisories include VMSA-2021-0002, VMSA-2024-0013, VMSA-2025-0004 (exploited in the wild) and VMSA-2025-0013 for ESXi, and VMSA-2023-0023, VMSA-2024-0019 and VMSA-2025-0010 for vCenter. ESXi checks are version-aware: the fix line is selected by the host's version and update level (build numbers are not monotonic across trains — e.g. 8.0 U3f 24784735 < 8.0 U2e 24789317); a train the tool doesn't track is graded verify, and ESX 9.x hosts are pointed at the VMSA rather than guessed.

vCenter CVEs are conservative advisories. vCenter exposes only a build number, and 7.0/8.0 build ranges overlap, so the verdict is: patched only if the build is at/above the highest known fix, review if below the lowest, else verify. Always confirm against the linked VMSA and your exact release. Fix builds are dated to the manual's build (2026-07) and should be spot-checked over time.

8. Fleet mode

Point the Bundle field at a parent folder and click Triage fleet folder…. The tool discovers every bundle recursively (extracted trees + .tgz/.tar), skips outer wrappers that already have an extracted sibling, runs the full pipeline per host, and builds a roster sorted worst-first: host, cluster, type, build, VMs, key-file coverage, logons/actions (flagged), persistence, indicators, CVE, and a composite risk score. Click any host to open its full single-host triage; the orange ◀ Fleet roster button returns you to the roster (no re-scan).

9. Clusters

ESXi hosts are grouped by their vSAN Sub-Cluster UUID (from vsan-cluster-get), which also names all cluster members. A vCenter is tied to a cluster when it manages a member host (from vpxd's host-N,<fqdn> map) or shares task opIDs with one. Nodes with no vSAN membership are labelled standalone. This lets a mixed folder of cluster nodes + standalone hosts + a vCenter self-organise.

10. Correlation & the vCenter→ESXi chain

The Correlation button on the roster opens cross-host analysis (computed from the in-memory snapshots — no re-parse):

  • Clusters — members present vs named-but-absent.
  • Source IPs across hosts — an IP on >1 host suggests lateral movement / shared attacker infra.
  • Accounts on multiple hosts — credential spread.
  • IOC hits across hosts and coordinated indicators (the same tell on N hosts = a mass action).
  • vCenter→ESXi task chain — an opID shared between a vCenter vpxd task and the host vpxa that executed it, reconstructing "which vCenter action ran on which host".
  • Unified cross-host flagged-event timeline, merged and time-ordered.

12. Caveats & limitations

Guest IPs are not in ESXi bundles (they are guest-side). The VMs tab shows a Guest IP column only when a vCenter in the same fleet reported them and the VM names match. In single-bundle mode the column is hidden.
vCenter task→user attribution is approximate — tasks are attributed to the nearest-preceding human SAML login, not an exact session-to-user binding. Treat "user approx" as a lead, corroborate with the Logons tab.
The vCenter→ESXi chain needs a vCenter and ESXi bundle from the same cluster in the fleet; different environments correctly show no shared opIDs.
CVE verdicts are advisory (see §6). Support bundles are targeted, not disk images — some persistence artefacts (cron, systemd-admin, authorized_keys, passwd) are simply not captured and are shown as "not captured".

13. Suggested DFIR workflow

  1. Scope: open the bundle (or fleet). Read the Overview — indicator panel, CVE panel, key-file coverage. On a fleet, sort the roster by risk.
  2. Know the node: Specification for config, networking, and (for recovery) vSAN/disk layout; VMs for the guest inventory.
  3. Access: Logons — external IPs, failed→success bursts, privileged and off-hours logins.
  4. Actions: what was done — shell commands (ESXi) / attributed vpxd tasks (vCenter). Watch the red shell command and SUSPECT/DESTRUCTIVE rows.
  5. Persistence & tampering: boot scripts, syslog forwarding, ransom-note scan, log-integrity gaps.
  6. IOCs: drop your IOC.txt in and re-analyse to sweep the logs.
  7. Fleet: Correlation for shared IPs/users/IOCs, coordinated indicators, and the vCenter→ESXi chain. Export the cross-host timeline.
  8. Document: Case-note report per host and/or the Fleet report for the engagement.
